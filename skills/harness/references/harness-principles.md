# Harness Engineering 원칙

OpenAI의 Codex 팀이 에이전트만으로 100만 줄 규모 제품을 구축하며 발견한 원칙들.

## 1. AGENTS.md는 목차다

- 거대한 지침 파일은 실패한다: 컨텍스트 희소, 지침 과부하, 빠른 부패
- 100줄 이하의 짧은 맵 + 구조화된 docs/ 디렉터리
- 기계적 검증 가능: 커버리지, 신선도, 소유권, 교차 링크

## 2. 리포지터리 = 기록 시스템

- 에이전트가 실행 중 접근할 수 없는 것은 존재하지 않는 것
- Google Docs, Slack, 사람 머릿속 지식은 시스템에서 불가
- 버전 관리되는 아티팩트(코드, 마크다운, 스키마, 실행 계획)만 유효
- 팀 결정은 반드시 repo에 기록

## 3. 아키텍처를 기계적으로 강제

- 커스텀 린터 + 구조적 테스트로 레이어 의존성 검증
- 에러 메시지에 수정 지침을 주입 (에이전트 프롬프트)
- 중앙에서 경계를 설정하고 로컬에서 자율성 허용
- 보통 수백 명 규모에서 하는 걸 에이전트 환경에선 초기에 해야 함

## 4. 점진적 공개

- 에이전트에게 1000페이지를 한번에 주지 않음
- 작고 안정적인 진입 지점에서 시작하여 깊이 탐색
- docs/ 디렉터리에 index.md로 색인화

## 5. 가비지 컬렉션

- 에이전트는 기존 패턴을 복제함 (불균일/최적 아닌 것도)
- 정기적 정리 프로세스 필수
- "황금 원칙"을 repo에 인코딩하고 주기적으로 편차 검사
- 기술 부채는 이자가 쌓이기 전에 조금씩 갚기

## 6. 앱 가독성

- 에이전트가 앱을 직접 실행하고 검증할 수 있게 만들기
- 로그, 메트릭, 추적을 에이전트에 노출
- worktree별 격리된 앱 인스턴스

## 7. 계획은 일급 아티팩트

- 실행 계획을 repo에 버전 관리
- active/, completed/, tech-debt-tracker.md
- 에이전트가 외부 상황에 의존하지 않고 작업 가능

## 8. "지루한" 기술 선호

- 에이전트가 추론하기 쉬운 안정적 API, 단순한 추상화
- 불투명한 업스트림보다 직접 구현이 더 싸울 수 있음
- 결합성, API 안정성, 학습 데이터 내 표현이 기준
