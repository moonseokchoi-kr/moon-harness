# Harness Scoring Guide

12개 원칙별 채점 기준. audit 시 이 문서를 참조하여 일관된 점수를 부여한다.

## 점수 산출 공식

```
DimA = (P1 + P2 + P5 + P12) / 4    # Documentation & Navigation (30%)
DimB = (P3 + P4 + P10) / 3          # Enforcement & Consistency (30%)
DimC = (P6 + P9 + P11) / 3          # Architecture & Knowledge (20%)
DimD = (P7 + P8) / 2                # Operations & Maintenance (20%)

종합 = (DimA×0.3 + DimB×0.3 + DimC×0.2 + DimD×0.2) × 10
```

등급: L1(0~19) / L2(20~39) / L3(40~59) / L4(60~79) / L5(80~100)

---

## P1. Agent Entry Point (에이전트 진입점)

CLAUDE.md 또는 AGENTS.md가 프로젝트 이해의 명확한 시작점인가?

**필수 포함:** 프로젝트 목적, 기술 스택, 디렉터리 구조, 빌드·테스트 명령, 핵심 규칙, 하위 문서 포인터

| 점수 | 기준 |
|------|------|
| 0 | 진입점 파일 없음, README도 에이전트 관점 정보 부재 |
| 3 | README에 기본 명령어만, 구조 설명 없음 |
| 5 | CLAUDE.md 존재하나 불완전하거나 오래됨 |
| 8 | 진입점 완비, 하위 디렉터리별 안내 부족 |
| 10 | 루트+하위 CLAUDE.md 모두 있고 정기 갱신 |

## P2. Map, Not Manual (지도, 매뉴얼 아님)

필요한 정보를 2단계 이내에 찾을 수 있는 구조인가?

**필수 요소:** 목차/앵커 링크, 문서 간 교차 참조, 단일 주제 집중, 검색 친화적 용어

| 점수 | 기준 |
|------|------|
| 0 | 문서 거의 없거나 순차적 매뉴얼 형태 |
| 3 | 기본 문서 있으나 긴 단일 파일 |
| 5 | 주제별 분리되나 상호 참조 부족 |
| 8 | 목차, 참조, 분리 모두 갖춤 |
| 10 | 어떤 질문든 2단계 탐색으로 답 발견 |

## P3. Invariant Enforcement (규칙 자동 강제)

코드 스타일, 아키텍처 규칙이 도구(linter, CI, hook)로 자동 강제되는가?

**대상:** 코드 스타일, 타입 검사, 테스트, 아키텍처 규칙, 명확한 에러 메시지

| 점수 | 기준 |
|------|------|
| 0 | 도구 없음, 모든 규칙이 구두/문서 의존 |
| 3 | 기본 도구 있으나 CI에서 선택적 |
| 5 | CI 필수이나 아키텍처 규칙은 수동 리뷰 |
| 8 | 핵심 invariant 도구 강제 + 명확 메시지 |
| 10 | 모든 규칙 자동화 + hook + 수정 가이드 포함 에러 메시지 |

## P4. Convention Over Configuration (관례 우선)

파일명, 디렉터리, API 규칙이 명시적이고 일관적이어서 패턴 복제만으로 올바른 코드가 나오는가?

| 점수 | 기준 |
|------|------|
| 0 | 규칙 없음, 혼합 스타일 |
| 3 | 일부 일관성, 전체 통일 부재 |
| 5 | convention 존재하나 명시적 문서 없음 |
| 8 | 문서화 + 도구 강제 + 대부분 일관적 |
| 10 | 문서+도구+코드생성기. 패턴 복제만으로 정확 코드 생성 |

## P5. Progressive Disclosure (단계적 정보 공개)

정보가 개요→모듈→심화 3단계 계층으로 적절히 분배되는가?

| 점수 | 기준 |
|------|------|
| 0 | 정보 계층 없음, 모든 것이 루트에 |
| 3 | 루트 + 일부 모듈 문서만 |
| 5 | 2단계 존재, ADR 없음 |
| 8 | 3단계 + 모듈별 문서 + ADR |
| 10 | 완전한 progressive disclosure, 모든 깊이에서 과부족 없음 |

## P6. Layered Architecture (계층형 아키텍처)

레이어 의존성이 단방향이고 도구로 강제되는가?

| 점수 | 기준 |
|------|------|
| 0 | 레이어 개념 없음, 의존성 무질서 |
| 3 | 디렉터리 분리만, 순환 의존성 존재 |
| 5 | 레이어 구분 + 대부분 올바른 방향 |
| 8 | 명확 구조 + 문서 + 도구 강제 |
| 10 | 완전 계층화, 순환 의존성 제로, 자동 강제 |

## P7. Garbage Collection (주기적 정리)

dead code, stale 문서, 방치된 TODO가 주기적으로 정리되는가?

| 점수 | 기준 |
|------|------|
| 0 | dead code, stale 문서 대량 |
| 3 | 일부 @deprecated만, 실제 정리 없음 |
| 5 | 주기적 정리 가끔, 자동화 없음 |
| 8 | 탐지 도구 + 정기 정리, TODO에 이슈 링크 |
| 10 | 자동화된 탐지 + CI 경고 |

## P8. Observability (관찰 가능성)

에이전트가 작업 결과를 빠르게 자체 검증할 수 있는가?

| 점수 | 기준 |
|------|------|
| 0 | 테스트 없음, 빌드 확인 방법 불명 |
| 3 | 테스트 존재하나 느림(10분+), 부분 실행 불가 |
| 5 | 테스트+type check 가능, 모듈 단위 어려움 |
| 8 | 모듈 단위 + type check + lint 빠르고 명확 |
| 10 | 초 단위 피드백, 파일 단위 테스트, 증분 빌드 |

## P9. Knowledge in Repo (레포 내 지식 저장)

프로젝트 이해에 필수적인 지식이 repo 안에 있는가?

| 점수 | 기준 |
|------|------|
| 0 | 대부분 팀원 머릿속 |
| 3 | 기본 설정 문서만, 아키텍처/비즈니스 규칙은 외부 |
| 5 | 주요 문서 있으나 일부 맥락 외부 |
| 8 | ADR + 규칙 문서 + 트러블슈팅 가이드, 일부 암묵적 지식 |
| 10 | 레포가 single source of truth, 외부 참조 불필요 |

## P10. Reproducibility (재현성)

동일 입력 → 동일 결과가 보장되는가?

| 점수 | 기준 |
|------|------|
| 0 | lock 파일 없음, "내 컴퓨터에서는 되는데" |
| 3 | lock 파일 있으나 환경 설정 수동 |
| 5 | 의존성 고정 + 기본 자동화, 일부 테스트 외부 의존 |
| 8 | 결정론적 빌드 + 컨테이너 + 대부분 테스트 격리 |
| 10 | 완전 재현성, 어떤 머신에서든 동일 결과 |

## P11. Modularity (모듈성)

각 모듈이 독립적으로 이해·수정·테스트 가능한가?

| 점수 | 기준 |
|------|------|
| 0 | 모듈 개념 없음, 수천 줄 단일 파일 |
| 3 | 파일 분리만, 경계 불분명 |
| 5 | 구조 존재, 대부분 독립적, 일부 강결합 |
| 8 | 명확 경계 + API 정의 + 독립 테스트 |
| 10 | 완전 모듈화, 변경 영향 항상 예측 가능 |

## P12. Self-Documentation (자기 설명적 코드)

코드 자체(이름, 타입, 구조)가 의도와 동작을 설명하는가?

| 점수 | 기준 |
|------|------|
| 0 | `a`, `tmp`, `data` 이름, 타입 없음, 매직 넘버 |
| 3 | 어느 정도 서술적이나 불일치 많음 |
| 5 | 대부분 서술적, 타입 부분적, "왜" 주석 부족 |
| 8 | 서술적 이름 + 강 타입 + 상수 + "왜" 주석 |
| 10 | 코드 자체가 완전 문서, 이름·타입·구조·주석 완벽 |

## 우선 개선 순서 (ROI 기준)

audit 리포트의 개선 항목은 이 순서로 정렬한다:

| 순위 | 원칙 | 이유 |
|------|------|------|
| 1 | P3 Invariant Enforcement | 도구 강제로 에이전트 오류율 극적 감소 |
| 2 | P1 Agent Entry Point | 초기 탐색 시간 90% 단축 |
| 3 | P8 Observability | 자체 검증으로 결과 품질 보장 |
| 4 | P4 Convention | 패턴 매칭 극대화 |
| 5 | P9 Knowledge in Repo | 지식 접근 가능 |
| 6 | P10 Reproducibility | 시행착오 감소 |
| 7 | P11 Modularity | 안전한 수정 |
| 8 | P6 Layered Architecture | 부작용 방지 |
| 9 | P12 Self-Documentation | 이해도 향상 |
| 10 | P2 Map, Not Manual | 탐색 효율 |
| 11 | P5 Progressive Disclosure | 정보 과부하 방지 |
| 12 | P7 Garbage Collection | 거짓 신호 제거 |
